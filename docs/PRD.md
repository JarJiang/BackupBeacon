# BackupBeacon 产品需求文档（PRD）

## 1. 背景

团队同时维护多个项目数据库，需要稳定、自动化、可审计的备份体系。人工备份容易遗漏，且执行不及时。

## 2. 产品目标

提供一个集中式内网备份平台，用户完成连接与策略配置后，系统按计划自动执行备份并记录全过程。

## 3. 当前范围（已冻结）

### 3.1 包含范围

1. 多连接管理（首期支持 MySQL/PostgreSQL）
2. 备份策略管理
   - 全量备份
   - 增量/追加备份（按数据库引擎能力实现）
3. 调度方式
   - 固定间隔（v1 必做）
   - Cron 表达式（v1.1 再补）
4. 备份目标写入服务器目录（支持共享磁盘挂载路径）
5. 任务历史与执行日志
6. 站内通知（失败/超时等）

### 3.2 不包含范围（当前阶段）

1. 恢复/回档流程
2. 外部通知（Telegram/Webhook）
3. 公网部署能力
4. 内置鉴权与 RBAC
5. 多容器复杂拆分

## 4. 核心用户故事

1. 作为运维人员，我可以新增多个数据库连接并验证连通性。
2. 作为运维人员，我可以按策略配置需要备份的库/表范围。
3. 作为运维人员，我可以设置备份间隔并让系统自动执行。
4. 作为运维人员，我可以把备份输出到服务器共享目录。
5. 作为运维人员，我可以在页面查看任务记录和失败通知。

## 5. 功能需求

### 5.1 连接管理

- 新增/编辑/删除数据库连接
- 新增连接采用默认收起 + 抽屉式交互（保持列表页面简洁）
- 抽屉内提供连接测试、保存连接、关闭操作
- 保存前连接测试
- 凭证存储（v1 先明文，后续可加密）

### 5.2 备份策略

- 全量/增量模式选择
- 库级/表级范围配置
- 备份目录（仅服务器目录视角）

### 5.3 调度系统

- 按分钟间隔调度
- 支持手动立即执行
- 同一策略避免重复并发

### 5.4 执行流水线

1. 读取策略
2. 组装备份命令
3. 执行导出
4. 写入目标目录
5. 更新任务记录
6. 失败写入站内通知

### 5.5 页面通知

- 失败任务显示为未处理通知
- 支持手动标记已处理

## 6. 非功能需求

- 通过 Docker Compose 单容器部署
- 本地可直接运行调试，不必每次打镜像
- 备份目录必须可写
- 最小网络暴露面（仅内网）

## 7. 实时进度

### 7.1 已完成

- 单容器架构落地（Spring Boot + SQLite + 静态前端）
- 基础页面与接口已可用
- 备份执行主流程可跑通
- JDK 8 环境锁定（本地与 Docker）
- 仓库首版已推送

### 7.2 进行中

- 连接测试接口完善
- 备份目录校验逻辑
- 页面中文提示与错误文案优化
- 目标目录文案动态化（容器路径优先，不写死宿主机路径）
- 可配置路径映射展示（`BACKUPBEACON_PATH_DISPLAY_MAP`）
- 立即备份 loading 与执行状态反馈
- 连接描述字段 + 上次备份时间字段（已完成）

### 7.3 待办（v1 后续）

- 策略编辑/删除
- 日志筛选与分页
- 配置导入导出

## 8. 验收标准（v1）

1. 连接管理默认收起新增表单，点击“新增连接”后通过抽屉完成配置。
2. 抽屉内“连接测试 / 保存连接 / 关闭”流程可用。
3. 能配置至少 5 个数据库连接。
4. 能配置至少 10 条备份策略。
5. 手动执行与自动执行都能产出备份文件。
6. 失败任务会在页面通知中可见。
7. 单容器可通过 `docker compose up -d --build` 启动。
